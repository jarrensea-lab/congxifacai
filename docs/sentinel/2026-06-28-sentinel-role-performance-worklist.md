## 1. 工作边界

本 work list 是 Sentinel 的绩效评审层，不直接生成买入、卖出、清仓或满仓指令。

评分参考见：

```text
docs/sentinel/2026-06-28-sentinel-role-scoring-reference.md
```

它评估的是：

- 四名辩论员的预测是否准确。
- 裁判的综合判断是否有效。
- 给用户的投资建议是否带来正向结果。
- 哪些角色需要提高、降低或限制权重。
- 每个角色是否按照自己的职责行事，而不是简单判断谁对谁错。

它不做：

- 不替代风控模块。
- 不自动调整真实仓位。
- 不自动修改角色 prompt 或模型配置。
- 不把历史成功率当作未来收益保证。

## 2. 阶段一：预测留痕

- [x] 定义角色预测记录格式。
- [x] 记录猎手的短线方向、技术信号、关注标的和置信度。
- [x] 记录账房的基本面判断、估值判断、低中频方向和置信度。
- [x] 记录守夜人的风险等级、风险事件、止损/观望理由和置信度。
- [x] 记录 Serenity 辩手的产业链主题、瓶颈判断、候选标的和置信度。
- [x] 记录裁判的最终结论、风险等级、仓位建议、采用/否决哪些角色观点。
- [x] 每条预测记录原始报告路径、生成时间、预测周期和输入数据摘要。

建议输出：

```text
data/sentinel/role_predictions/YYYY-MM-DD.jsonl
```

验收标准：

- 每次日报、盘前或收盘辩论后，都能回查每个角色当时说了什么。
- 每条记录都有 `role`、`prediction_type`、`target`、`horizon`、`confidence`、`source_report`。
- 预测记录只保存研究和评估字段，不保存密钥、cookie、token 或 webhook。

## 3. 阶段二：真实结果对账

- [x] 定义预测结果回查格式。
- [x] 对指数方向预测进行回查。
- [x] 对板块方向预测进行回查。
- [x] 对个股涨跌预测进行回查。
- [x] 对风险提醒进行回查，区分风险命中、误报和漏报。
- [x] 对观望建议进行回查，判断是否避免亏损或错过收益。
- [x] 对关注建议进行回查，判断是否跑赢基准。

建议输出：

```text
data/sentinel/role_outcomes/YYYY-MM-DD.jsonl
```

验收标准：

- 能按 1 日、3 日、5 日、20 日周期回看预测。
- 能区分未到期预测、已验证预测和无法验证预测。
- 对缺失行情或停牌等情况标记 `unavailable`，不伪造结果。

## 4. 阶段三：角色评分

- [x] 计算每个角色的方向准确率。
- [x] 计算每个角色的收益误差。
- [x] 计算每个角色的风险误报率。
- [x] 计算每个角色的风险漏报率。
- [x] 计算置信度校准度：高置信度判断是否真的更准。
- [x] 计算稳定性：最近 5 日、20 日、60 日表现是否持续。
- [x] 计算职责履行评分：是否按照角色职能给出有证据的判断。
- [x] 将用户执行纪律从角色评分中拆出来单独归因。
- [x] 记录不够精准的原因：信息不足、调查不深、逻辑错误、过度乐观、过度保守或市场突发。
- [x] 单独计算裁判最终建议的综合评分。

建议输出：

```text
data/sentinel/role_scores/YYYY-MM-DD.json
```

验收标准：

- 每个角色都有滚动评分。
- 分数能解释原因，不只给一个数字。
- 低样本量时必须显示样本不足，不给过度结论。
- 角色建议合理但用户未执行时，不扣角色分。

## 5. 阶段四：投资建议成功率

- [x] 区分纸面建议和真实执行建议。
- [x] 记录用户是否实际执行了建议。
- [x] 记录执行时间、价格、数量和持有周期。
- [x] 记录用户执行是否符合建议的价格、仓位、时间和风险条件。
- [x] 记录未执行或偏离执行的原因。
- [x] 计算执行建议的胜率。
- [x] 计算执行建议的平均收益。
- [x] 计算执行建议的最大回撤。
- [x] 计算建议是否跑赢沪深 300、中证 2000 或对应行业指数。
- [x] 计算建议是否改善账户风险，而不只看短期涨跌。

建议输出：

```text
data/sentinel/advice_performance/YYYY-MM-DD.json
```

验收标准：

- 能回答“哪些建议我真的执行了，结果如何”。
- 能回答“哪些建议只是纸面预测，不能算真实收益”。
- 能回答“建议本身是否合理”和“用户是否按建议执行”。
- 报告中明确区分收益、风险降低和错失机会。

## 5A. 阶段四补充：用户执行纪律评分

- [x] 定义用户执行纪律记录格式。
- [x] 标记未执行、部分执行、延迟执行、超仓位执行、偏离价格执行。
- [x] 区分用户主动选择不执行和客观条件无法执行。
- [x] 对执行偏离造成的盈亏单独归因。
- [x] 在报告中明确说明执行纪律不计入角色评分。

建议输出：

```text
data/sentinel/execution_discipline/YYYY-MM-DD.json
```

验收标准：

- 角色评分和用户执行纪律评分分开展示。
- 角色建议合理但用户未执行，不扣角色分。
- 用户偏离建议导致亏损，不直接归因给角色。

## 6. 阶段五：角色调整建议

- [x] 识别最近失准的角色。
- [x] 识别过度乐观或过度保守的角色。
- [x] 识别对特定行情更擅长的角色。
- [x] 识别裁判是否过度依赖某个角色。
- [x] 输出角色权重调整建议。
- [x] 输出 prompt 调整建议。
- [x] 输出需要人工复核的问题。

建议动作类型只允许：

```text
increase_weight
decrease_weight
keep_weight
watch_role
review_prompt
require_human_review
```

验收标准：

- 调整建议必须引用评分证据。
- 调整建议不能直接变成自动买卖。
- 任何权重调整都需要用户确认后才能进入配置。

- [x] 生成每周完整角色评分报告。
- [x] 生成投资建议绩效报告。
- [ ] 报告写入司库归档。
- [ ] 飞书只推摘要，不推长表格。

建议输出：

```text
sentinel_role_scorecard.md

```

报告必须包含：

- 角色排名。
- 样本数量。
- 准确率。
- 收益表现。
- 风险误报和漏报。
- 真实执行建议表现。
- 下阶段角色调整建议。

## 8. 阶段七：接入现有流程

- [x] 在日报生成时保存角色预测。
- [x] 在盘前策略生成时保存角色预测。

- [ ] 在周报任务中生成角色评分。

验收标准：

- 有评分数据时，裁判报告能看到“历史表现提示”。
- 历史表现只能作为参考，不能绕过当日事实和风控。

验收标准：

## 10. 推荐执行顺序

```text
第 1 步：先做预测留痕。
第 2 步：补真实结果对账。
第 3 步：做角色评分。
第 4 步：做投资建议成功率。
第 5 步：生成角色评分报告。
第 6 步：再把评分摘要旁路接入裁判。
第 7 步：最后考虑权重和 prompt 调整建议。
```

## 11. 优先级

P0 必须先做：

- 预测留痕。
- 真实结果对账。
- 角色评分。
- 纸面建议和真实执行建议区分。

P1 随后做：

- 投资建议成功率。
- 角色评分报告。
- 角色调整建议。

P2 再做：

- 裁判旁路引用历史评分。
- 飞书摘要。
- 长周期模型/prompt 调整建议。
