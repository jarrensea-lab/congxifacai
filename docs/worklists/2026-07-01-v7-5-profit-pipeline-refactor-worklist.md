# v7.5 Profit Pipeline Refactor Worklist

> 目标：把“研究型日报”重构为“账户操作策略 + 标的生命周期评分 + 触发预警 + 复盘自迭代”的盈利导向主线。

## 0. 核心原则

- 第一屏永远先回答账户操作：持仓怎么处理、卖不卖、买不买、多少钱、什么价格、错了哪里止损、今天不动明天等什么信号。
- 空仓只是账户状态，不是默认策略立场。
- 买不起的股票不能进入生产可执行标的池，只能进入研究参照池。
- “数据不足，建议观望”禁止作为泛化兜底。只有接口真实失败时才能写数据缺失；否则必须写具体原因：信号未触发、赔率不足、一手金额超限、资金流未确认、公告催化不足、追高风险。
- Sentinel/Serenity/辩论细节单独归档在司库相关目录；主报告只引用结论、评分、证据链接和可执行动作。
- 所有模块必须服务三个目标：盈利、清晰策略、自行迭代。

## 1. 当前结构性问题

### 1.1 报告引擎

- 当前报告先写“明日总策略”，后面又重新从“市场概况/持仓概览/AI分析/辩论结论”编号，结构重复。
- 角色辩论原文过长，占用主报告注意力，但没有优先回答账户动作。
- “AI 关注标的”混合了可执行标的和买不起的研究标的，容易误导。
- “知识角”适合单独学习归档，不应挤占投资策略主报告。
- DeepSeek/Qwen 健康状态仍显示 `unknown`，没有接真实运行状态。

### 1.2 标的池

- 当前 `data/candidate_pool.json` 同时承载研究线索、观察标的和可交易标的，状态边界不清。
- Sentinel/Serenity 候选可直接进入生产池，导致 `688008/002371/688012` 这类一手金额超账户能力的标的进入交易主线。
- 标的池缺少入池前账户可执行性过滤。
- 标的池缺少明确出池规则：逻辑证伪、风险升高、连续不达标、流动性不足。

### 1.3 Sentinel

- Sentinel 能锁定热点、主题、风险事件，但目前入池动作过宽。
- Sentinel 研究包应该成为归一证据源，不应该直接决定可交易标的池。
- Sentinel 输出需要区分：主题方向、风险事件、标的提及、可交易候选、研究参照。

### 1.4 Serenity

- Serenity 深挖产业链有价值，但当前在辩论里容易重复讲产业链报告。
- Serenity 应从“自由辩手”改为“产业链证据审计员”。
- Serenity 必须补一个输出：如果龙头买不起，给同产业链可交易替代标的。
- Serenity 参与辩论的权重应按策略类型变化：短线 15%-20%，中线 25%-35%，产业主题确认有否决权，买入执行无单独决定权。

### 1.5 数据源引擎

- 实时行情、K线、成交额、资金流、北向、新闻公告类、财务指标大多已经能抓。
- 问题不是“没有数据”，而是没有把数据统一压成每只股票的每日评分卡。
- 数据源引擎现在没有统一产出 `TargetDataSnapshot`，导致辩论仍然依赖长文本和模型自行理解。

### 1.6 辩论与裁判

- 辩论现在直接读市场摘要和研究文本，缺少结构化标的评分输入。
- 裁判会输出“空仓观望但启动准备”这种模糊结论，不够可执行。
- evidence_id 有被模型编造的风险，必须只能从 evidence ledger 注入。
- 账户约束执行过晚，应该在入池前、辩论前、报告前重复生效。

### 1.7 监控与预警

- 持仓预警有基础能力，但观察池触发价、放量、资金流转强、公告催化的飞书预警还不够。
- 当前飞书推送以报告为主，缺少“事件触发型机会/风险提醒”。

### 1.8 自迭代

- 角色预测和 scorecard 已有雏形，但没有反向修正下一次角色权重、评分阈值、入池/出池。
- 每天像重新写一份报告，不像在维护一套会进化的策略系统。

## 2. 新目标架构

```text
账户状态
  -> Sentinel 发现方向/风险/主题
  -> Serenity 产业链确认与替代标的
  -> Evidence Ledger 证据归一
  -> Target Pool 分层
  -> Target Data Snapshot 补全数据
  -> Target Scoring Engine 评分
  -> Account Executability Gate 账户可执行性过滤
  -> Debate 只围绕评分卡争论
  -> Judge 输出 buy / sell / hold / watch / remove
  -> Report Engine 决策优先展示
  -> Feishu Trigger Alerts
  -> 1/3/5/20 日复盘回填
  -> 权重和阈值修正
```

## 3. 标的池分层

### 3.1 `executable`

- 买得起最小交易单位。
- 满足账户单票预算。
- 有明确买入触发价、止损价、目标价。
- 流动性和成交额通过。
- 风险收益比通过。

### 3.2 `watching`

- 买得起，但信号未触发。
- 必须写清楚等待条件：价格、量比、成交额、资金流、公告、指数环境。

### 3.3 `research_reference`

- 产业链重要，但账户买不起或不适合交易。
- 只作为板块锚点和研究参照。
- 不进入今日买入建议。

### 3.4 `removed`

- 逻辑证伪、风险升高、连续不达标、流动性差、公告利空、题材退潮。

## 4. 报告引擎新结构

### 4.1 第一屏：今日账户操作策略

固定回答：

```text
1. 当前持仓怎么处理？
2. 是否需要卖？
3. 如果卖，卖哪只？
4. 卖多少？
5. 看什么卖出信号？
6. 是否需要买？
7. 如果买，买哪只、买多少、什么价格买？
8. 如果今天不动，明天等什么信号？
```

### 4.2 第二屏：持仓策略

即使空仓，也要写现金部署策略，不写“空仓先入为主”。

字段：

- 当前持仓。
- 持仓动作：持有、减仓、清仓、止损、止盈。
- 卖出触发：跌破、放量破位、冲高回落、公告利空、资金流转弱。
- 卖出数量。
- 止损价。
- 止盈价。
- 盘中预警。

### 4.3 第三屏：新开仓策略

字段：

- 今日是否开仓。
- 买入标的。
- 买入金额。
- 买入价格/触发价。
- 止损价。
- 目标价。
- 仓位。
- 触发条件。
- 不买原因。

### 4.4 第四屏：标的池

分四张表：

- 今日可执行标的。
- 观察等待触发标的。
- 研究参照标的。
- 今日剔除标的。

### 4.5 第五屏：数据覆盖与评分

每只股票展示：

- 实时行情。
- 历史 K 线。
- 成交额/换手/量比。
- 个股资金流。
- 北向/市场资金环境。
- 新闻/公告。
- 财务指标。
- Sentinel 证据。
- Serenity 产业链证据。
- 账户可执行性。

### 4.6 第六屏：复盘与自迭代

- 昨日建议表现。
- 3 日建议表现。
- 5 日建议表现。
- 20 日建议表现。
- 哪个角色判断正确。
- 哪个角色误判。
- 权重如何调整。

### 4.7 单独归档

以下内容从主报告移出，保存到司库相关目录：

- 完整辩论记录。
- Sentinel 归一数据包。
- Serenity 深挖产业链瓶颈报告。
- 角色 scorecard。
- 原始 evidence ledger 明细。

主报告只保留链接和摘要。

## 6. 验收标准

- 报告第一屏 30 秒内能回答：买、卖、不动、价格、金额、止损、目标、明日触发条件。
- 买不起的股票不出现在今日可执行买入建议中。
- 每只股票有评分卡，不再泛化写“数据不足，建议观望”。
- Sentinel/Serenity 细节单独归档，主报告只保留结论和链接。
- 飞书能推送持仓风险和观察池机会。
- 复盘结果能影响下一次角色权重或评分阈值。
- 全量测试通过：`PYTHONPATH=.:backend .venv/bin/python -m pytest backend/tests/ -q`。
- Lint 通过：`.venv/bin/python -m ruff check backend scripts`。

## 7. 暂停条件

- 数据源真实不可用，且无法确认是否是权限、网络、接口变更问题。
- 账户规则需要用户重新定义，例如单票预算、最大亏损、是否允许科创板。
- 自动预警会触发真实飞书推送前，必须确认是否允许。
- 任何模块需要使用未提供的新凭证。
